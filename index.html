CallBack & Promise

A callback function is a function that is passed as an argument to another function and is executed after some operation has been completed.  it is  asynchronous, non-blocking I/O model, enabling developers to handle delayed operations, such as database queries or file reads.

Error Handling: Hard to catch errors when execution is externally controlled.

You give control of when your callback executes to another function :- It means you're delegating the timing and execution of your code to external code. You write a function, but you don't call it yourself - you hand it over, and someone else decides when and how to run it.

Asynchronous Execution: Callbacks are often used to handle asynchronous operations, allowing the program to continue running without blocking.

Higher-Order Functions: A function that takes another function (callback) as an argument is called a higher-order function.

Callback Hell (Pyramid of Doom): Multiple nested callbacks can lead to code that is hard to read and maintain.

Error Handling: In Node.js, callbacks often follow the error-first pattern, where the first argument is an error object.

Promises  provide a simpler, more organized, and synchronous way to handle asynchronous operations. A and Promise is a JavaScript object that represents the eventual completion (or failure) of an asynchronous operation

  • With Promises, you attach callbacks (with .then() or .catch()) and the Promise guarantees that they will be called, but the timing is still controlled by the Promise.     • Promise API

    • .then(): Chains actions for when a promise is resolved. Takes two optional arguments: one to handle the resolved value and one for rejection.

    • .finally(): Executes a callback after a promise is settled (resolved or rejected).

    • Pending: Initial state. The promise's outcome is yet undecided.
    • Fulfilled (Resolved): The operation was successful. The promise has a resolved value.
    • Rejected: The operation encountered an error. The promise is rejected and has a reason for rejection.

Map/Set

A Map is a collection of key-value pairs where each key and value can be of any data type. Unlike the simpler object literal, Map allows any data type to be used as a key, and it maintains the order of the keys. Here's a basic usage example:

// Creating a Map
let myMap = new Map();

// Adding key-value pairs
myMap.set('key1', 'value1');
myMap.set(42, 'value2');
myMap.set({ key: 'obj' }, 'value3');

// Accessing values
console.log(myMap.get('key1')); // Output: value1

// Checking if a key exists
console.log(myMap.has(42)); // Output: true

// Deleting a key-value pair
myMap.delete(42);

// Iterating through the Map
myMap.forEach((value, key) => {
console.log(`${key}: ${value}`);
});

Shallow Cloning/Deep Cloning:

Shallow Cloning

Shallow cloning creates a new object with the same top-level structure as the original object. if the original object contains nested objects or arrays, those nested structures are still referenced in the new object. Shallow cloning does not create entirely independent copies of nested structures.

const originalObject = { a: 1, b: { c: 2 } };

const shallowClone = Object.assign({}, originalObject);

----------------------------------------------

const originalObject = { a: 1, b: { c: 2 } };

const shallowClone = { ...originalObject };

const originalArray = [1, 2, 3, { a: 4 }];

// Shallow clone using Array.from

const shallowClone = Array.from(originalArray);

console.log(shallowClone);

const originalArray = [1, 2, 3, { a: 4 }];

 

// Shallow clone using slice

const shallowClone = originalArray.slice();

 

console.log(shallowClone);

 

Deep Cloning:

Deep cloning creates a new object with an entirely independent copy of all nested structures. The new object is entirely separate from the original object, and changes in one do not affect the other.

 

const originalObject = { a: 1, b: { c: 2 } };

const deepClone = JSON.parse(JSON.stringify(originalObject));

 

Using a Deep Cloning Library (e.g., Lodash):

const _ = require('lodash');

const deepClone = _.cloneDeep(originalObject);

Type Coercion in JavaScript refers to the automatic conversion of values from one data type to another.

  3. Conditionals (if and Ternary Operators): Truthiness or falsiness is determined.

  4. Logical Operators: Non-boolean values are coerced to booleans.

    • Example: "hello" && 0 evaluates to 0 because the truthy "hello" short-circuits the && operation, and 0 coerces to false.

Strict equality

Strict equality (===) in JavaScript requires both value and type to match, testing for more specific conditions and reducing the likelihood of unexpected results.

In contrast, the abstract equality comparison (==) can lead to type coercion, potentially causing counterintuitive outcomes.

While both comparison modes test value equality, === ensures an additional match of data type.

Illustrative Example: Abstract vs. Strict Equality

  • Abstract Equality:
    • 5 == '5' evaluates to true because JavaScript converts the string to a number for comparison.
  • Strict Equality:
    • 5 === '5' evaluates to false because the types are not the same.

Key Considerations

  • Type Safety: === is safer as it avoids unwanted type conversions.
  • Performance: === can be faster, especially for simple comparisons, as it doesn't involve type coercion or additional checks.
  • Clarity: Favoring === can make your code clearer and more predictable.

Common Best Practices

  • Use Strict Equality by Default: This approach minimizes unintended side effects.
  • Consider Type Coercion Carefully: In specific cases or with proven understanding, == can be suitable, but be cautious about potential confusion.

Consider a simple scenario of a function in charge of generating a secret password:

  return function getPassword() {
      return password;
  };
}

Arrow functions

Arrow functions in JavaScript are a concise way to write anonymous functions. They provide a more concise syntax compared to traditional function expressions.

Key differences and features of arrow functions:

Conciseness: Arrow functions are often more concise than traditional functions, especially when dealing with simple, single-expression functions.

Implicit Return: If the arrow function body consists of a single expression, it is implicitly returned without the need for the return keyword.

No Binding of this: Arrow functions do not bind their own this value. Instead, they inherit the this value from the enclosing execution context (lexical scoping).

No arguments Object: Arrow functions do not have their own arguments object. If you need to access the arguments, useing the rest parameters (...) or use the arguments object of the enclosing scope.

 

Arrow Functions: Key Features

  • Concise Syntax:

    • Especially useful for short, one-liner functions.
    • No need for function keyword or braces if there's a single expression.

  • Implicit Return:

    • When there's no explicit { return ... ;} statement, arrow functions return the result of the single expression inside.

  • this Binding:

    • Does not have its own this. It's "inherited" from the surrounding (lexical) context. This feature is known as 'lexical scoping'.

Code Example: Standard Function vs. Arrow Function

Here is the JavaScript code:

// Standard Function
function greet(name) {
  return "Hello, " + name + "!";
}

// Arrow Function
const greetArrow = name => "Hello, " + name + "!";

In the code above, greet is a standard function, while greetArrow is an arrow function, showcasing the difference in syntax and required keywords.

When to Use Arrow Functions

  • Event Handlers: Ideal for concise, inline event handling, where this context can be inherited from the lexical scope.

  • Callback Functions: Useful for array methods like map, filter, and reduce.

  • Avoidance of this Redefinition: When you want to maintain the surrounding context of this and avoid unintended redefinition.

  console.log('Button clicked:', this);  // Refers to the global/window object
};

In the arrow function example, the context of this does not refer to the button element, but to the global window object, because arrow functions do not have their own binding of this. Instead, they inherit this from their lexical scope, which in this case is the global context.

encapsulating private state

In JavaScript, encapsulating private state within an object can be achieved using a closure. This ensures the state is local to the object and not directly accessible from outside.

How Closures Work

A closure allows a function to retain access to the lexical environment (the set of variable bindings at the point of function declaration) in which it was defined, even when the function is executed outside that lexical environment.

This means that any inner function, defined inside another function, has access to the outer function's variables, and that access is maintained even after the outer function has finished executing.

For example:

function outerFunction() {
    let outerVar = 'I am outer';  // This variable is in the lexical environment of outerFunction

    function innerFunction() {
        console.log(outerVar);  // Accesses outerVar from the lexical environment of outerFunction
    }

    return innerFunction;
}

let myInnerFunction = outerFunction();
myInnerFunction();  // Logs: "I am outer"

Here, innerFunction retains access to outerVar.

When defining a JavaScript constructor function with function and new, closure can be used to associate private state with each instance:

function Gadget() {
    let secret = 'top secret';
    this.setSecret = function (value) {
        secret = value;
    };
    this.getSecret = function () {
        return secret;
    };
}

let phone = new Gadget();
phone.setSecret('new secret');
console.log(phone.getSecret());  // 'new secret'

In this example, secret is private to each Gadget instance, thanks to closure.

Modules

In modern JavaScript, module patterns combined with immediately-invoked function expressions (IIFE) are often used for encapsulation and data hiding.

  • The revealing module pattern enables selective exposure of private members.

  • The IIFE pattern immediately executes and returns the object to be assigned, effectively creating a module.

Here is the code:

let myModule = (function () {
    let privateVariable = 'I am private';

    function privateMethod() {
        console.log('I am a private method');
    }

    return {
        publicMethod: function () {
            console.log('I am a public method');
        },
        getPrivateVariable: function () {
            return privateVariable;
        }
    };
})();

console.log(myModule.getPrivateVariable());  // 'I am private'
myModule.privateMethod();  // Throws an error because privateMethod is not exposed

In this example, privateVariable and privateMethod are accessible only within the IIFE's lexical environment, thus making them private.

Prototypes

Prototypes in JavaScript are objects from which other objects inherit properties. They're the foundational concept behind JavaScript's object-oriented approach and can be leveraged for efficiency and resource sharing.

Core Concept

  • Each constructor function has a unique prototype object.
  • Objects instantiated from the constructor function link to this prototype.
  • If an object doesn't have a property, JavaScript looks up the prototype chain.

Visual Representation

The .__proto__ property connects instances to their constructor's prototype, forming a prototype chain.

Here is the code representation:

function Person(name) {
  this.name = name;
}

let person1 = new Person('Alice');
let person2 = new Person('Bob');

console.log(person1.__proto__ === Person.prototype); // Output: true

Inheritance Mechanism

Prototype-based inheritance facilitates the reuse of behaviors and methods among objects.

Here's an illustration:

// Define a "Person" constructor
function Person(name) {
  this.name = name;
}
// Augment its prototype with a common method
Person.prototype.introduce = function() {
  console.log(`Hello! I'm ${this.name}.`);
};

// Create instances and call the shared method
let person1 = new Person('Alice');
let person2 = new Person('Bob');

person1.introduce(); // Output: "Hello! I'm Alice."
person2.introduce(); // Output: "Hello! I'm Bob."

// Illustration of how the prototype mechanism works
console.log(person1.__proto__ === Person.prototype); // Output: true
console.log(person1.__proto__.introduce === Person.prototype.introduce); // Output: true

Prototypes & Constructors

In JavaScript, objects can be created from functions acting as constructors and classes.

  • Constructor Functions: Such functions create objects with given properties and a direct link to their own prototype.
  • Classes: Introduced in ECMAScript 6 with syntax like class myClass {...}. They are essentially complex constructor functions, providing a more intuitive way to work with object-oriented JavaScript.

Pitfalls with Prototypes

  1. Shared State: Mutations to an inherited entity are visible across all derived objects.

  2. Concurrency Issues: If an inherited state is mutable and not synchronized, it poses risks in multi-threaded scenarios.

  3. Dynamic Link: Mutating an object's prototype affects all subsequent references, which can be error-prone in complex systems.

Practical Applications

  • Code Reusability: Reusing methods defined on the prototype is efficient, especially for large numbers of objects.
  • Efficient Memory Use: When an instance doesn't require its own copy of a function, it can share it via the prototype, saving memory.

Best Practices

  • Avoid Modifying Built-in Object Prototypes: It can lead to unexpected behavior and compatibility issues.
  • Use Prototypes for Efficient Resource Sharing: Store shared methods or unchanging, frequently accessed data on the prototype.

Prototypal inheritance

Prototypal inheritance is a fundamental concept in JavaScript that underpins how objects inherit and delegate behavior.

Key Components

  1. Central Role of __proto__: Every JavaScript object has this hidden property, pointing to another object. This 'another object' is the object it inherits properties and methods from.

  2. Mechanisms for Property Access:

    • hasOwnProperty: A method attached to the object's prototype.
    • Object.getOwnPropertyDescriptors(): Provides detailed information about an object's properties.

Historical Perspective

  • Influences from Lisp and Self: JavaScript's unique inheritance model draws inspiration from these languages, especially Self, which is a dynamically-typed, prototype-based language.

Code Example: Prototypal Inheritance

// Create the parent object
let vehicle = {
  intro: 'I am a vehicle.',
  sound: function() {
    return 'Vrrrooom!';
  }
};

// Create the child object
let car = Object.create(vehicle);
car.wheels = 4;

// Modify a method in the child object
car.sound = function() {
  return 'Vroom vroom!';
};

console.log(car.sound());  // Outputs: Vroom vroom!

// Object property search path (order of lookup):
// car -> vehicle.__proto__ (or vehicle) -> Object.prototype (last resort)

console.log(car.hasOwnProperty('intro'));  // Outputs: false
// The 'intro' property is not a direct property of car but is instead inherited from vehicle.

console.log(Object.getOwnPropertyDescriptors(car));
// Outputs detailed property information about car, showing 'sound' as a regular property and 'intro' as an inherited one.

Best Practices

  • hasOwnProperty Check: Useful when working with inherited properties to distinguish between an object's own properties and those inherited from its prototype chain.
  • Direct Property Definitions: For modern browsers, use methods like Object.defineProperty() for precision in property definition.

Pitfalls

  • Shared References: Prototypical inheritance promotes reference sharing, which can lead to unexpected modifications in a shared object.

  • Dynamic Prototype Linkages: Changes to an object's prototype are immediately reflected in all its derived objects.

async/await

Async

Async simply allows us to write promises-based code as if it was synchronous and it checks that we are not breaking the execution thread.

Async functions will always return a value. It makes sure that a promise is returned and if it is not returned then JavaScript automatically wraps it in a promise which is resolved with its value.

Await

Await function is used to wait for the promise. It could be used within the async block only.

Mechanism of async/await

When a function is asynchronous, it can use the await keyword within to pause its execution until a Promise resolves or rejects.

Under the hood, await checks if the value is a Promise. If it is, the function interally uses then and catch to wait for the Promise to settle.

Improving Readability

Code written with async/await is typically more readable and easier to follow than its Promise-based counterpart. It often resembles synchronous, linear code, which makes it easier to write, debug, and maintain.

Here is a side-by-side example of a Promise-based function and its equivalent with async/await to highlight the differences in readability between the two methods:

Promise-Based

function fetchFirstUser() {
  return fetch('/users')
    .then(response => response.json())
    .then(users => users[0])
    .then(user => fetch(`/users/${user.id}/profile`))
    .then(profileResponse => profileResponse.json())
    .then(profile => console.log(profile))
    .catch(error => console.error('Something went wrong:', error));
}

async/await

async function fetchFirstUser() {
  try {
    const response = await fetch('/users');
    const users = await response.json();
    const user = users[0];
    const profileResponse = await fetch(`/users/${user.id}/profile`);
    const profile = await profileResponse.json();
    console.log(profile);
  } catch (error) {
    console.error('Something went wrong:', error);
  }
}

In the async/await version, each step is more clearly defined, and error handling feels more naturally integrated.

Error Handling

Both Promises and async/await offer ways to handle errors, but the syntactic differences can lead to variations in readability.

  • Promises: Errors are generally handled in the final .catch() block or within each .then() block.
  • async/await: A try...catch block is used to catch and handle all errors in one place, providing a more centralised approach to error management.

Event propagation

Event propagation in the Document Object Model (DOM) refers to the mechanism where events are processed and passed through nested elements. There are two stages in event propagation.

Event Phases

  • Capturing Phase: Events start from the top-most ancestor and flow down to the event target. This phase is less common in practice.

  • Bubbling Phase: Events begin at the event target and move up through ancestors, triggering event listeners set on those elements.

Key Concepts

  • Target Element: The element on which the event was first recognized.

  • Current Element: The element which is currently being evaluated for event handlers.

  • Event Listener: A function designed to receive and process events.

  • Event Object: Contains information relevant to the event, such as the type of event and the target element.

  • Stop Propagation: A method that allows event handlers to interrupt propagation, either capturing or bubbling, of the event.

Practical Applications

  • Event Delegation: Utilizes the bubbling phase to monitor events on parent elements, optimizing memory and performance.

  • Dynamic Content Handling: For content added post-page load, propagation ensures event listeners are attached and functioning as expected.

Code Example: Event Propagation

Here is the JavaScript code:

// Using capturing phase
document.getElementById("parent").addEventListener("click", handleClick, true);
document.getElementById("child").addEventListener("click", handleClick, true);

// Using bubbling phase (default)
document.getElementById("parent").addEventListener("click", handleClick);
document.getElementById("child").addEventListener("click", handleClick);

To see event propagation in action, here is the additional HTML:

<div id="parent">
  <div id="child">Click me!</div>
</div>

Each div has relevant styles and handleClick can console log to visually confirm event propagation.

Best Practices

  • Consistency: Choose an event phase—either bubbling or capturing—and stick to it for all event handling to avoid confusion.

  • Performance: Leverage bubbling for improved speed, especially with larger web pages.

  • Legacy Systems: Keep in mind older systems and libraries that might rely on capturing to avoid compatibility issues.

  • Middlewares & Frameworks: Some tools, e.g., frameworks like Angular, have explicit preferences for event propagation. Ensure you're aligned with these requirements for seamless integration.

Life Cycle Considerations

Certain actions, such as unloading a page or removing an element from the DOM, should prompt the removal of associated event listeners to prevent memory leaks. Failure to do so can result in unexpected page behavior and performance degradation.

Modern development frequently involves anonymous listeners, where functions are defined and utilized solely for their capacity as event handlers. This approach supports the maintenance of clean, encapsulated code.

Consistent patterns and meticulously-applied rules for both erecting and dismantling listeners contribute to a codebase's maintainability and robustness.

Event handlers in JavaScript can be tricky to navigate, especially when it comes to understanding the behavior of this within their context.

Mechanism of 'this' in Event Handlers

Every time a function is invoked, its this value is determined. For event handlers, the invocation happens as a response to a specific user interaction, such as a button click.

The value of this within these event triggers depends on the context or the way it is bound.

Global Context

Here is the JavaScript code snippet:

  console.log(this); // Refers to the button (DOM element) that fired the click event
});

When invoked in a global context, this typically refers to the window object. However, when such a function is executed as an event handler, its this is set to the element that triggered the event.

Bound Context

Developers can control the context of functions using methods like bind, call, or apply. This allows them to force a particular object as context for function execution.

In the following code, this inside printPerson refers to the person object, even though it's triggered by a button click:

  lastName: 'Doe',

};

  console.log(this); // Refers to the button (DOM element) that fired the click event
  console.log(person.fullName());
}.bind(person));

Be aware that arrow functions handle this differently and do not have the typical concept of this binding.

For most typical event-driven interactions, such as DOM events like mouse clicks or keyboard inputs, the binding of this within the event handler function becomes crucial for successful handling and interaction management.

Cross-Origin Resource Sharing (CORS)

  • Cross-Origin Resource Sharing (CORS) is a mechanism that allows web servers to specify which domains are permitted to access their resources. This is a crucial security measure to guard against unintended data access.

    Introducing Same-Origin Policy

    Websites operate within a same-origin policy, which states that they can only access resources (e.g., API endpoints) from the exact same protocol, hostname, and port number.

    Cross-Origin Resource Sharing (CORS) acts as a security gatekeeper, laying out specific conditions for one web domain to interact with another, relaxing the default restrictions of the same-origin policy.

    How CORS Works

    1. Request Initiation: When a web page requests a resource from another domain (the "foreign domain"), the browser first sends a "preflight" request, using the HTTP method OPTIONS, to determine if the foreign domain permits access.

    2. Server Response: Upon receiving the preflight request, the foreign server must acknowledge with specific CORS headers. If the headers permit access, the browser forwards the original request.

      • Access-Control-Allow-Origin: This header lets the server specify which origins it's willing to share the resource with. Common values include a specific origin (e.g., https://example.com) or the wildcard * to permit any origin.
      • Additional Headers: If the web page tries to send non-standard headers or specifies HTTP methods other than the more common ones (like GET or POST), the server must specify which of these are allowed using Access-Control-Allow-Headers and Access-Control-Allow-Methods headers, respectively.

    *. If the request doesn't meet the CORS criteria, the server responds with an error status code (for example, 403 Forbidden).

    *. If the response includes Access-Control-Allow-Origin: *, this indicates that the server allows access to its resource from any origin, making it less secure.

    1. Browser Action: If the server responds with the necessary headers allowing the request, the browser forwards the original request to the server. If the server is satisfied with the parameters of this request (for example, the origin or the headers), it responds with the actual resource.

    Common HTTP Verbs

    • Simple Methods: These include GET, POST, and HEAD. They are considered safe for read-only operations and do not trigger a preflight request under CORS.
    • Preflight Methods: Any method not in the list of simple methods, e.g., PUT or `DELETE, requires a preflight request.

    Practical Use-Cases

    • RESTful API Integration: A server hosting a RESTful API might specify that certain HTTP methods and headers from origins other than its own are allowed.
    • Cross-Domain Data Access: For instance, host websites might allow access to their data from particular partner domains. E-commerce websites could integrate external payment gateways by permitting CORS requests.

Content Security Policy (CSP)

The Content Security Policy (CSP) is a layered security measure that serves as a gatekeeper for the resources a webpage can load and execute. By controlling what's permissible, CSP plays a crucial role in thwarting various security attacks.

Mechanisms for Content Control

  • Source Whitelisting: CSP permits specifying the origins from which certain content can be loaded, including media, scripts, and stylesheets. This whitelist approach reduces the risk of content served from unanticipated sources.

  • Directives for Granular Control: CSP offers a range of directives, each tailored to regulate a specific type of content. You can fine-tune the permissions for styles, scripts, and frames independently to attain the desired balance between flexibility and security.

  • Reporting Mechanisms and Violation Reports: When a policy is breached, CSP furnishes detailed reports. This feature is particularly valuable during the initial stages of policy adoption, providing insights into potential false positives and aiding iterative policy refinement.

Security Benefits

  • Prevention of XSS Attacks: By inhibiting the execution of unauthorized scripts, CSP acts as a shield against cross-site scripting (XSS), a prevalent attack vector to inject and execute malicious scripts within the context of a different page.

  • Minimization of Code Injection: CSP's mandate for script sources from explicitly defined locations adds a layer of defense against arbitrary code injections, characteristic of injection attacks like cross-site request forgery (CSRF) or script injection attacks.

  • Safeguarding Against Data Interception: The directive to only permit encrypted media through HTTPS lessens the risk of privacy invasions through unencrypted content.

Fallback Mechanisms

  1. Upgrade-Insecure-Requests: This automatic HTTP-to-HTTPS upgrade mechanism streamlines the transition to secure browsing by transforming HTTP requests into their HTTPS counterparts.

  2. Block-All-Mixed-Content: CSP's capability to suppress mixed content ensures that secure pages aren't at the mercy of insecure resources, averting potential security loopholes.

Implementations Across Technologies

  • Apache HTTP Server: Integrate CSP via mod_headers module with .htaccess.
  • Nginx: Employ add_header to include CSP directives in server responses.

  • WordPress: Leverage the right plugin, such as "WP Content Security Policy".
  • Drupal: Utilize modules like "Security Kit" to configure CSP directives.

A design pattern in JavaScript is a proven, reusable solution to a specific type of problem in software design that has been formalized to assist developers in solving common challenges. JavaScript has some common patterns that are extensively used across the industry.

Types of Design Patterns in JavaScript

  • Creational: Deal with object creation mechanisms, providing ways to create objects in a manner suitable for the given situation.

  • Structural: Focus on object composition, managing relationships between objects, and making them more compatible.

  • Behavioral: Define manners in which different objects and classes interact. They help ensure a wide range of flexibility and communication between them.

Examples of Design Patterns in JavaScript:

Creational Patterns

  1. Constructor Pattern: This is similar to a class. JavaScript objects, including complex structures, are created explicitly with a constructor.

  2. Module Pattern: Offers a way to boost encapsulation. It leverages closures to define private variables and exposes a public API to interface with those variables.

  3. Factory Pattern: Acts as a 'factory'. It provides a method for creating objects without specifying the class of the object.

  4. Singleton Pattern: Ensures that only a single instance of a class is present, and that instance is made globally accessible.

Structural Patterns

  • Decorator Pattern: Dynamically adds or alters behavior in an object, often an instance of another class. It's useful for extending objects without the need to create a new subclass.

  • Facade Pattern: Offers a simplified interface to a more complex system of classes and functionalities. It can be thought of as a "front" for a collection of services or subsystems.

  • Adapter Pattern: Acts as a bridge between two incompatible interfaces. It's useful for making two classes work together that otherwise wouldn't because of differing interfaces.

Behavioral Patterns

  • Observer Pattern: Establishes a one-to-many dependency relationship between objects, ensuring that when a subject changes state, all its dependents, or observers, are notified and updated automatically.

  • Strategy Pattern: Defines a family of algorithms, encapsulates each one, and makes them interchangeable. It allows the algorithm to vary independently from the clients that use it.

  • Iterator Pattern: Provides a way to access the elements of an aggregate object sequentially without exposing its underlying representation.

  • Mediator: Defines an object that encapsulates how a set of objects interact. Mediator promotes loose coupling by keeping objects from referring to each other explicitly.

  • Chain of Responsibility: Pattern to pass a request along a chain of handlers. The request is sent from one object to the next across the chain until an object effectively handles it.

Others

  • Memory Pattern: Focuses on limiting memory consumption in applications. For example, caching strategies and avoiding memory leaks can be described as memory design patterns.

  • Integration Design Pattern: Provides broad, abstract ideas for integrating systems. For example, front-end/back-end communication and techniques are part of integration design patterns.

The Revealing Module Pattern is an advanced way to encapsulate and organize JavaScript code, which brings the benefits of closure and object literal notation together.

This pattern utilizes a distinctive coding style that makes it clear which of the module's functions and variables are public, while keeping all others private.

Key Components

  1. Closures: Inner functions inherit the scope of their parent.

  2. Self-Invoking Function: The module code self-executes and returns an object or function to the outer environment, ensuring the module's privacy.

  3. Object Pack: The module uses an object to group both private (closures) and public members.

Module Structure

The module begins with an IIFE, immediately invoked to execute the code and return the object containing the public functions.

The code components inside this IIFE are separated into public functions that will be attached to the returning object and private functions and variables that won't be exposed.

Code Example: Revealing Module Pattern

Here is the JavaScript code:

const revealingModule = (function() {
  // Private variables and functions
  let privateVar = 'I am private';

  function privateFunction() {
    console.log('This function is private and not accessible outside the module.');
  }

  // Public variables and functions, revealed upon module return
  return {
    publicVar: 'I am public',

    getPrivateVar: function() {
      return privateVar;
    },

    setPrivateVar: function(value) {
      privateVar = value;
    },

    // Reveal private function for public use
    publicFunction: function() {
      privateFunction();
    }
  };
})();

// Example using the module
console.log(revealingModule.publicVar); // Output: I am public
console.log(revealingModule.getPrivateVar()); // Output: I am private
revealingModule.setPrivateVar('Changed!');
console.log(revealingModule.getPrivateVar()); // Output: Changed!
revealingModule.publicFunction(); // Output: This function is private and not accessible outside the module.

Advantages

  • Privacy Control: Private members aren't accessible from outside the module, enhancing code security.

  • Singleton Creation: Even though it's possible to create multiple instances, the pattern commonly results in a singleton-like module where a single instance is returned.

Cautions

  • Live Updates: As JavaScript objects and functions are passed by reference, altering a public object or function that references a private member allows indirect access.

  • Testing Difficulty: The approach can pose challenges in unit testing private methods, potentially making them harder to assess.

Recursive functions

Recursive functions offer a way to solve problems wherein a solution depends on solving simpler instances of the same problem.

In the context of computing factorials, the term "recursive" implies the factorial n is calculated in terms of the factorial of n−1.

Factorial Function Example

Here is the JavaScript code:

function factorial(n) {
    if (n === 0) {
        return 1;
    }
    return n * factorial(n - 1);
}

Execution Example: factorial(3)

Here is the execution:

  1. factorial(3) = 3 * factorial(2)
  2. factorial(2) = 2 * factorial(1)
  3. factorial(1) = 1 * factorial(0)
  4. factorial(0) reached, returns 1

Unrolling the stack:

  • factorial(0) evaluated to 1, so factorial(1) evaluated to 1 * 1 = 1
  • factorial(1) evaluated to 1, so factorial(2) evaluated to 2 * 1 = 2
  • factorial(2) evaluated to 2, so factorial(3) evaluated to 3 * 2 = 6
  • Final result: factorial(3) = 6

async Deep

The async keyword in JavaScript marks a function as asynchronous, telling the interpreter to execute it on a separate event loop. This enables the use of two additional keywords: await and return. It plays a key role in asynchronous programming using promises.

Underlying Mechanism

Using the async keyword instructs JavaScript to do the following when the function is called:

  1. Return a Promise: If your function explicitly returns a result, the async function will wrap it in a resolved Promise. If the function doesn't return an explicit value, it will implicitly return a resolved Promise undefined.

  2. Handle Promise Rejections: The async function will automatically reject the Promise if an error is thrown or an exception is produced.

  3. Enable await Usage: The await keyword can only be used within an async function. It instructs JavaScript to pause the execution of the function until the awaited Promise resolves.

Example: async and await

Here is the JavaScript code:

function resolveAfter2Seconds() {
  return new Promise(resolve => {
    setTimeout(() => {
      resolve('resolved');
    }, 2000);
  });
}

async function asyncCall() {
  console.log('calling');
  let result = await resolveAfter2Seconds();
  console.log(result);
  console.log('called');
}

asyncCall();

In this example, when asyncCall is invoked, the message "calling" is printed first. Then, the function waits for 2 seconds until the Promise returned by resolveAfter2Seconds resolves. The await keyword instructs the code to pause execution.

Once the promise resolves, the value 'resolved' is printed, and 'called' is printed after it. The console.log(result) statement indeed waits for the resolveAfter2Seconds promise to resolve.

Document Object Model (DOM)

  • The Document Object Model (DOM) is an essential concept in web development, where it serves as a programming interface for web documents. It provides a structured representation of the HTML document, enabling language-agnostic access and manipulation. The DOM forms the bridge between web content and the various programming languages working behind the scenes.

    Key DOM Components

    1. Document: Represents the web page, serving as the entry point to the DOM and offering methods for document-wide functionality.

    2. Element: Represents the HTML elements within the document, providing access to their attributes and contents, and allowing for manipulation.

    3. Node: Acts as the overarching interface for each item in the DOM tree. It defines fundamental methods and attributes applicable to all types of nodes, including elements, text, comments, and document nodes.

    DOM Building Blocks

    • Nodes: All DOM elements form a hierarchical tree composed of individual nodes. Each node is an object within the document with unique sets of characteristics.

    • Elements: Nodes that correspond to specific HTML tags, such as <p> for paragraphs and <a> for hyperlinks.

    • Attributes: Elements possess attributes (e.g., id, class, href) that can be queried and modified dynamically.

    • Text Nodes: Captures the text within an element, such as the words inside a <p> tag.

    Document Object

    The Document Object provides entry points to the entire document and serves as a global reference for numerous DOM features. It's universally accessible via the document variable in JavaScript. The object offers methods for selecting elements through CSS and XPath queries, among other functionalities.

    1. Interactive UIs: The DOM is the foundation for any interactive web application. It allows for event handling and real-time content updates.

    3. Dynamic Content: Websites can generate content dynamically, altering the DOM to cater to user actions or data updates.

Dom created, modified, and interacted

The DOM (Document Object Model) presents a structured representation of an HTML document, which can be created, modified, and interacted with through code.

Key Methods

  • Create: document.createElement()
  • Append: parentNode.appendChild() or parentNode.insertBefore()
  • Remove: nodeToRemove.parentNode.removeChild(nodeToRemove) & parentNode.removeChild(childNode).

Code Example: Create and Append a New Element

Here is the JavaScript code:

// Create a new paragraph element
let newParagraph = document.createElement('p');

// Set its text content
newParagraph.textContent = 'This is a dynamically created paragraph!';

// Find a container (e.g., div with id 'container') to append the paragraph to
let container = document.getElementById('container');

// Append the paragraph to the container (at the end)
container.appendChild(newParagraph);

You can also insert an element before an existing one (as a reference point) using parentNode.insertBefore(newNode, existingNode)

Code Example: Remove an Element

Here is the JavaScript code:

// Find the element to remove (e.g., element with id 'toBeRemoved')
let toBeRemoved = document.getElementById('toBeRemoved');

// Remove the element from the document
toBeRemoved.parentNode.removeChild(toBeRemoved);

Code Example: Clear All Child Elements of a Parent

Here is the JavaScript code:

// Find the parent element (e.g., element with id 'parentElement')
let parentElement = document.getElementById('parentElement');

// Clear all its child elements

    parentElement.removeChild(parentElement.firstChild);

Considerations and Best Practices

  • Minimize DOM Manipulations: Heavy and frequent DOM manipulations can lead to slow, unresponsive web pages, especially in older browsers. Whenever possible, group changes together and use techniques such as fragment caching for performance benefits.
  • Live vs Static NodeLists: The returned collections from queries like getElementsByClassName are live, while those from querySelectorAll are static. A live collection auto-updates with the DOM, reflecting any subsequent changes.
  • Event Delegation: Instead of individually attaching event listeners to multiple child elements, favor delegating events to their common parent. This is particularly useful when the child elements are dynamic or arrive later on the page. Libraries like jQuery provide a convenient on() method for event delegation. For native JavaScript, use parentNode.addEventListener() with careful event target selection (event.target).

innerHTML and textContent

When using innerHTML and textContent in JavaScript, you primarily operate on the content of an HTML element. The distinction emerges in their handling of text nodes, HTML tags and formatting.

Key Distinctions

  • Form of Output:

    • innerHTML: Renders content with HTML, allowing for styling and tags.
    • textContent: Renders content as a string, displaying tags as literal text.

  • Content Sanitization:

    • innerHTML: Requires careful handling to avoid security risks like XSS attacks.
    • textContent: Safer for text input as it treats all content as literal text.

  • Performance Considerations:

    • innerHTML: Slower due to potential parsing and formatting overhead.
    • textContent: Generally faster as it's a direct text assignment.

Code Example: innerHTML vs textContent

Here is the JavaScript code:

// Using innerHTML
document.getElementById('myDiv').innerHTML = '<p>Hello, <strong>World!</strong></p>';

// Using textContent
document.getElementById('myDiv').textContent = '<p>Hello, <strong>World!</strong></p>';

In the first example, the content is dynamically formatted with p and strong tags, visible in the rendered HTML. In the second example, the content is displayed as a plain string of text on the webpage, including the HTML tags as literal text.

Dom Memory-Efficient

Managing DOM events is critical to ensure performance and sidestep potential memory leaks. Let's look at some best practices for robust event handling.

Best Practices for Memory-Efficient Event Handling

  • Event Delegation: Rely on the document object or a parent node of the children that need event handling.

  • Element Removal: When you no longer need an element, uninstall any event listeners, and remove the element from the DOM to free up both memory and event resources.

  • Use Events Sparingly: Continuously sparking or listening to a vast number of events can be a drain on your application.

  • Batch Operations: Try to consolidate your event handling on a per-frame basis for efficiency. This could be particularly useful during tasks such as dragging and dropping. The Browser might execute operations in batches, thus ensuring if you can consolidate your operations together could be more efficient.

Limitations of Batching

Most Browsers do batch operations, and this could be because of the way they are engineered. Using native event listeners might help to ensure that they are operating efficiently as per the browser's optimization techniques. However, to take the full benefit, consulting the specific documentation and using the most excellent techniques could be beneficial.

  • Plugin System: A rich ecosystem of plugins offers numerous utilities, such as bundle optimization.

Window object

The Window object in JavaScript serves as the global context for both the browser and Node.js. It provides access to global functions and variables and acts as the primary interface to the browser's user interface.

In browsers, the Window object is the top level of the browser's Document Object Model (DOM) tree. This grants access to various features, including the global document object, location information, and the setTimeout function.

Global Functions and Variables

The Window object provides a range of global functions and variables that can be accessed directly without listing Window as the object.

For example, common functions like alert(), prompt(), and confirm(), which are used to display various types of user messages, are all part of the Window object.

DOM Properties

When the JavaScript engine runs in a browser, Window is the global object, and properties of this object are accessible as global variables.

For instance, window.innerWidth retrieves the width of the browser's content area. Setting global variables, such as innerWidth, is actually modifying properties of the Window object.

Example: Using Window Properties

Here is the JavaScript code:

// Access and modify window properties directly
const height = window.innerHeight;
document.write('Current window height is: ' + height + ' pixels.');

// Update window height directly
window.innerHeight = 400;
document.write('Updated window height is: ' + window.innerHeight + ' pixels.');

Browser Methods

The Window object simplifies interaction with the browser through its methods. For example, you can use window.open() to launch a new browser window or tab:

// Open a URL in a new browser window or tab
window.open('https://www.example.com');

GlobalThis

To ensure global object consistency whether in web browsers or Node.js, the globalThis provides global access, and any property set through it is reflected in both contexts.

Node.js Context

In Node.js, the Window object is not present. Despite this, many built-in features still function as properties of the global global object, such as setInterval and timeouts.

Browser Compatibility

The Window object is specific to web browsers and is thus non-standard. But in practical settings, it provides vital global utilities.

Key Takeaways

  • The Window object is native to web browsers and not available in Node.js.
  • It serves as the global context for client-side JavaScript, providing access to the DOM, browser methods, and more.
  • Not all properties and methods of Window are cross-browser compatible, so it's essential to test functionality across different browsers.

Document Object Model (DOM)

The Document Object Model (DOM) is an in-memory representation of a web page that enables JavaScript to interact with it. The document object, derived from the DOM, serves as an entry point for such tasks.

Key Features

  • Element Selection and Modification: Allows direct actions on elements in the web page, like modifying their style or content.

  • Event Handling: Provides mechanisms for capturing and directing various events on the page.

  • Access to Other Objects: Allows access to objects like the browser history and location.

Browser Compatibility

While the majority of properties and methods are standard across modern browsers, there might be certain discrepancies. Tools like "caniuse.com" can assist in evaluating cross-browser compatibility.

Document Object Hierarchy

The document object acts as the root, providing access to different web page components through methods like getElementById() and properties like cookie.

Example: Directing Styles on an Element

Here is the JavaScript code:

document.getElementById("myElement").style.color = "red";

In this case, "myElement" is an ID within the web page, and style accesses its visual properties.

Querying Elements

The document object offers numerous methods to locate page elements using CSS selectors, tag names, or IDs:

  • getElementById()
  • getElementsByName()
  • getElementsByTagName()
  • getElementsByClassName()

Modern query methods such as querySelector() and querySelectorAll() use CSS selector patterns. Both return a live or static NodeList, respectively.

Usage Example: Select Forms

const formById = document.getElementById("myForm");
const firstInputByName = document.getElementsByName("inputName")[0];
const elementsByTag = formById.getElementsByTagName("h2");
const elementsByClass = document.getElementsByClassName("inputClass");
const firstInputByQuery = document.querySelector("input[type='text']");

Element Creation

The document object allows creating elements in-memory before inserting them into the web page through append or similar methods.
